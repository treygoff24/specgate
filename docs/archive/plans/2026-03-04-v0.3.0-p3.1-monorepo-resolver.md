# v0.3.0 P3.1: Full Monorepo Multi-Context Resolver

> **For Claude:** Spawn `task-builder` agents to implement this plan. Use `/orchestrator` for execution.

**Goal:** Make specgate explicitly monorepo-aware by wiring workspace discovery into the `check` pipeline, adding a configurable `tsconfig_filename` knob, adding batch workspace mode to the npm snapshot generator, and shipping integration + perf tests that prove it works at scale.

**Architecture:** The resolver already walks up directories to find the nearest `tsconfig.json` and pools `oxc_resolver::Resolver` instances per tsconfig path. This plan extends that by:
1. Plumbing a configurable tsconfig filename through `SpecConfig` → `ModuleResolver` → `nearest_tsconfig_for_dir_uncached`
2. Exposing discovered workspace packages in the check pipeline for diagnostic/telemetry purposes (not for file discovery — that stays filesystem-based per decision)
3. Adding `--workspace` batch mode to the npm snapshot generator
4. Adding integration fixtures + perf benchmarks

**Decisions Made:**
- Scope: Option C (Full Monorepo) — integration fixture + perf tests + config override + wire workspace discovery into check
- Source discovery: Option A (Leave filesystem-based) — no tsconfig-aware discovery
- npm snapshot: Option B (Batch mode) — `--workspace` flag for auto-discovering packages
- Config knob: Option A — single `tsconfig_filename` field (default: `"tsconfig.json"`)
- Version: v0.3.0 focused release

## Task 2: Create monorepo multi-tsconfig test fixture

**Parallel:** yes (batch 1) | **Blocked by:** none | **Owned files:** `tests/fixtures/monorepo-multi-tsconfig/**`

### Fixture structure

```
tests/fixtures/monorepo-multi-tsconfig/
├── package.json                    # { "workspaces": ["packages/*"] }
├── specgate.config.yml             # spec_dirs: ["packages"]
├── tsconfig.json                   # root: { "references": [...] }
├── packages/
│   ├── shared/
│   │   ├── tsconfig.json           # { "compilerOptions": { "composite": true, "paths": { "@shared/*": ["src/*"] } } }
│   │   ├── modules/
│   │   │   └── shared.spec.yml     # module: shared, path: "packages/shared/src/**/*"
│   │   └── src/
│   │       ├── index.ts            # export { greet } from './greet'
│   │       └── greet.ts            # export function greet() {}
│   └── web/
│       ├── tsconfig.json           # { "compilerOptions": { "composite": true, "paths": { "@web/*": ["src/*"], "@shared/*": ["../shared/src/*"] } } }
│       ├── modules/
│       │   └── web.spec.yml        # module: web, path: "packages/web/src/**/*", allow_imports_from: ["shared"]
│       └── src/
│           ├── index.ts            # import { greet } from '@shared/greet'
│           ├── app.ts              # import { helper } from '@web/helper'
│           └── helper.ts           # export function helper() {}
```

### Design notes
- Two packages with separate `tsconfig.json`, each with their own `paths` aliases
- `@shared/*` alias defined in both tsconfigs (web references shared)
- `web.spec.yml` declares `allow_imports_from: ["shared"]` so cross-package import is legal
- Root `tsconfig.json` uses project references (not needed for specgate, but realistic)
- Should also include a **violation case**: a file in `shared/src/` that imports from `@web/helper` (not allowed)

### Acceptance criteria
- Fixture exists and is self-consistent
- `specgate check` on this fixture produces a passing verdict when imports respect boundaries
- `specgate check` catches the violation when shared imports from web

**Parallel:** yes (batch 1) | **Blocked by:** none | **Owned files:** `npm/specgate/src/generate-resolution-snapshot.js`, `npm/specgate/bin/specgate-resolution-snapshot.js`, `npm/specgate/generate-resolution-snapshot.test.js`

```javascript
// In generate-resolution-snapshot.js
function discoverWorkspacePackages(projectRoot) {
    // 1. Check for pnpm-workspace.yaml
    //    Parse YAML, expand glob patterns under projectRoot
    // 2. Else check package.json "workspaces" field
    //    Expand glob patterns
    // 3. For each discovered package dir:
    //    - Find its tsconfig.json (or custom filename via --tsconfig-filename flag)
    //    - Return { name (from package.json), dir, tsconfigPath }
    // 4. Return [] if no workspace config found
    //
    // IMPORTANT: Cross-reference src/spec/workspace_discovery.rs for parity.
    // Must handle both pnpm-workspace.yaml and package.json workspaces with
    // the same glob expansion behavior as the Rust implementation.
}
```

#### 6b: Add batch snapshot generation

```javascript
function generateWorkspaceSnapshot(projectRoot, options) {
    const packages = discoverWorkspacePackages(projectRoot);
    if (packages.length === 0) {
        throw new Error('No workspace packages found');
    }

    const results = [];
    for (const pkg of packages) {
        if (!pkg.tsconfigPath) continue;
        const snapshot = generateSnapshot(pkg.dir, pkg.tsconfigPath, options);
        results.push({
            package: pkg.name,
            directory: pkg.dir,
            tsconfig: pkg.tsconfigPath,
            snapshot,
        });
    }

    return {
        snapshot_kind: "doctor_compare_tsc_resolution_batch",
        project_root: projectRoot,
        packages: results,
        generated_at: new Date().toISOString(),
    };
}
```
