SPECGATE DOGFOOD PLAN — HEARTH CODEBASE

Goal: Set up Specgate in the Hearth repo with prescriptive architectural specs,
run it against the current codebase, fix any violations, then wire it into CI
as a permanent enforcement gate.

===============================================================================
PHASE 1: INSTALL & INITIALIZE
===============================================================================

1. Install the specgate binary (already built at ~/Development/specgate):

   cp ~/Development/specgate/target/release/specgate /usr/local/bin/specgate

2. Initialize specgate in Hearth:

   cd ~/Development/hearth
   specgate init

   This creates specgate.config.yml and a modules/ directory.

3. Configure specgate.config.yml:

   spec_dirs:
     - "modules"
   exclude:
     - "node_modules"
     - "dist"
     - ".next"
     - "coverage"
   test_patterns:
     - "**/*.test.ts"
     - "**/*.test.tsx"
     - "**/*.spec.ts"

===============================================================================
PHASE 3: FIRST RUN — FIND VIOLATIONS
===============================================================================

Run specgate with no baseline (prescriptive mode):

   cd ~/Development/hearth
   specgate check

This will produce a JSON verdict listing every violation. Expect:
- Zero violations on the top-level boundaries (server↔client↔shared are clean)
- Possible violations on server internal boundaries (transport→watchers?)
- Possible violations on client internal boundaries (stores importing from
  somewhere unexpected, etc.)

Review the violation list. Each one is a refactoring task.

If the violation count is zero: congratulations, the architecture already
matches the ideal. Skip to Phase 5.

If violations exist: proceed to Phase 4.

===============================================================================
PHASE 4: FIX VIOLATIONS
===============================================================================

For each violation, fix the import to respect the boundary:

Common fixes:
- Move shared logic to support/ or shared/ if two modules both need it
- Use dependency injection (pass callbacks/interfaces) instead of direct imports
- Extract a type to shared/ if it's used across the server/client boundary
- If a store imports from a component, that's a real architecture bug — restructure

After fixing each batch, re-run `specgate check` to verify the violation count
drops. Repeat until clean.

===============================================================================
PHASE 5: GENERATE BASELINE (optional, for any deferred violations)
===============================================================================

If there are violations you intentionally defer (rare with prescriptive specs,
but possible):

   specgate baseline --output .specgate-baseline.json
   git add .specgate-baseline.json

This baselines those specific violations so they don't block CI while you
fix them in later PRs.

===============================================================================
PHASE 6: WIRE INTO CI
===============================================================================

Add to your GitHub Actions workflow (or create .github/workflows/specgate.yml):

   name: Specgate Architecture Check
   on: [push, pull_request]
   jobs:
     specgate:
       runs-on: ubuntu-latest
       steps:
         - uses: actions/checkout@v4
           with:
             fetch-depth: 0  # needed for blast-radius mode
         - name: Install Specgate
           run: |
             cargo install --locked --git https://github.com/treygoff24/specgate --tag v0.1.0-rc3
         - name: Check architecture boundaries
           run: specgate check
         - name: Check new violations only (PR mode)
           if: github.event_name == 'pull_request'
           run: specgate check --since origin/main --baseline-new-only

This gives you:
- Full check on every push (catches any violation)
- Blast-radius mode on PRs (only checks modules affected by the PR)
- Baseline-aware (deferred violations don't block, new ones do)

===============================================================================
PHASE 7: VALIDATE THE DOGFOOD
===============================================================================

After a week of running, evaluate:

1. Did specgate catch any real boundary violations in new work?
2. Was the onboarding experience smooth? (doc gaps, confusing errors?)
3. Did blast-radius mode work correctly on PRs?
4. Were there any false positives?
5. How does baseline drift look?

Feed findings back into Specgate issues/fixes.
